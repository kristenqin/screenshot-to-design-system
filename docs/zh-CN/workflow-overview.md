# 工作流总览

## 问题定义

直接让 AI 从截图生成设计稿，成功率低，因为它需要在一次生成里同时完成布局、颜色、字体、层级、组件结构和编辑能力。

更合理的方式是把任务拆成多个阶段，并让每个阶段都产生可以检查的中间结果。

更准确的产品表述是：

> 从视觉结果反推设计系统，再用设计系统重建页面。

## 端到端流程

```text
截图输入
  -> 图像预处理
  -> 线稿生成
  -> UI 元素检测
  -> OCR 和语义解析
  -> UI AST
  -> 相似组件提取
  -> design token 提取
  -> 设计系统生成
  -> 页面重建
  -> 渲染并与原图对比
  -> 根据 visual / structural diff 迭代
```

## 关键阶段

### 截图输入

用户应该提供尽量完整的软件截图。多状态、多页面、多设备截图会更有价值。

### 线稿生成

MVP 阶段继续使用豆包生图 API 作为主要线稿生成模块。

原因：

- 当前质量已经足够好。
- 自研成本高。
- 替代方案未必更便宜或更稳定。
- 这不是当前产品壁垒。

### UI AST

关键中间产物不是线稿图，而是结构化 UI AST。

它需要表达：

- 屏幕尺寸
- 元素 id
- 元素类型
- bbox
- 文本
- 样式
- 检测来源
- 置信度

### 组件候选

重复或相似区域先输出为 component candidates，而不是直接宣布生成最终组件。

这样可以保留人工或 AI review 空间，降低误判风险。

### 重建和反馈

生成结果需要重新渲染成图片，并和原截图对比。

反馈信号包括：

- pixel diff
- layout diff
- OCR text diff
- color distance
- bbox distance
- component count mismatch
